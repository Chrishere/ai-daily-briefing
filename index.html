OpenAI 工程文：在没有原生沙箱的 Windows 上，给 Codex 造一个"四层防御"安全沙箱

来源：openai.com · Building a safe, effective sandbox to enable Codex on Windows（5 月 14 日）
总结：当 Codex 工程师 David Wiesen 去年 9 月加入团队时，Codex 在 Windows 上没有 sandbox 实现。macOS/Linux 都自带沙箱原语，Windows 没有，于是 OpenAI 必须从零造一个。最终方案是双轨：PowerShell 用 Windows-native 沙箱（首选 "elevated" 模式），WSL2 里复用 Linux 沙箱实现。Elevated 模式靠"防御纵深"四层堆叠：专用低权限沙箱用户 CodexSandboxUsers、文件系统 ACL 限制可写目录、Windows 防火墙规则阻断出向网络、本地策略调整。这是 agent 安全工程的好教科书——agent 跑命令时怎么避免横向移动到开发者主会话。
查看英文原文摘录
If both modes are available, use elevated. The elevated sandbox is the preferred native Windows mode. It implements defence in depth through four layers: Dedicated sandbox user — a lower-privilege Windows account (CodexSandboxUsers group) that runs agent commands, preventing lateral movement from the agent process to the developer's session; Filesystem ACLs — permission boundaries restricting write access to the project working directory; Windows Firewall rules — blocking outbound network access from the sandbox user unless explicitly approved; and local policy changes needed for commands that run in the sandbox. On Windows, Codex uses the native Windows sandbox when you run in PowerShell and the Linux sandbox implementation when you run in WSL2.
查看中文翻译
如果两种模式都可用，请用 elevated（提权）模式。Elevated 沙箱是首选的 Windows-native 模式，它通过四层防御纵深实现：专用沙箱用户——一个低权限 Windows 账户（CodexSandboxUsers 组），用来运行 agent 命令，防止 agent 进程横向移动到开发者本人的会话；文件系统 ACL——把可写权限边界限制在项目工作目录之内；Windows 防火墙规则——除非显式批准，否则阻断沙箱用户的出站网络访问；以及沙箱内命令所需的本地策略调整。在 Windows 上，Codex 当你在 PowerShell 里运行时使用 Windows-native 沙箱，在 WSL2 里运行时使用 Linux 沙箱实现。
